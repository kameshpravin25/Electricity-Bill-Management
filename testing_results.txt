7.1 Testing Result of Sprint 1 (Database Testing)
--------------------------------------------------
Testing Type: Structural Testing
Database: PostgreSQL (Supabase Cloud)
Schema: 9 tables - users, customers, meters, tariffs, invoices, bills, payments, feedbacks

Test Cases Performed:

  Test ID | Test Case                                          | Expected Result                                      | Actual Result                                        | Status
  --------|----------------------------------------------------|----------------------------------------------------- |------------------------------------------------------|-------
  DB-01   | Create all tables using supabase_schema.sql         | All 9 tables created without errors                  | Tables created successfully in Supabase              | PASS
  DB-02   | Primary Key uniqueness on customers.id              | Auto-increment serial ID, no duplicates              | Each customer gets unique ID (1, 2, 3...)            | PASS
  DB-03   | Primary Key uniqueness on invoices.id               | Auto-increment serial ID, no duplicates              | Each invoice gets unique ID                          | PASS
  DB-04   | Foreign Key: meters.customer_id -> customers.id     | Meter must reference valid customer                  | Insert with invalid customer_id rejected             | PASS
  DB-05   | Foreign Key: invoices.customer_id -> customers.id   | Invoice must reference valid customer                | Referential integrity maintained                     | PASS
  DB-06   | Foreign Key: payments.invoice_id -> invoices.id     | Payment must reference valid invoice                 | Invalid invoice_id rejected                          | PASS
  DB-07   | Foreign Key: meters.tariff_id -> tariffs.id         | Meter must reference valid tariff                    | Referential integrity maintained                     | PASS
  DB-08   | CASCADE delete: Delete customer -> delete meters    | Deleting customer removes associated meters          | Meters deleted via ON DELETE CASCADE                 | PASS
  DB-09   | CASCADE delete: Delete customer -> delete invoices  | Deleting customer removes associated invoices        | Invoices deleted via ON DELETE CASCADE               | PASS
  DB-10   | CHECK constraint on users.role                      | Only 'admin' or 'customer' allowed                   | Insert with role='staff' rejected                    | PASS
  DB-11   | UNIQUE constraint on users.email                    | No duplicate email addresses                         | Duplicate email insert rejected                      | PASS
  DB-12   | NOT NULL constraint on users.email, users.password  | Null values rejected                                 | Insert without email/password fails                  | PASS
  DB-13   | Default values: invoices.status defaults to Pending | New invoice status = Pending                         | Default applied correctly                            | PASS
  DB-14   | Default values: invoices.amount_paid defaults to 0  | New invoice amount_paid = 0                          | Default applied correctly                            | PASS
  DB-15   | Seed data: 3 tariff plans inserted                  | Residential (Rs.5.50), Commercial (Rs.8), Industrial (Rs.12) | All 3 tariffs present in table              | PASS
  DB-16   | Sample data: 7 customers with meters, invoices      | All relational data inserted correctly               | 7 customers, 7 meters, 15 invoices, 12 payments     | PASS
  DB-17   | Data retrieval: JOIN customers + meters              | Customer-meter relationship queryable                | JOIN query returns correct paired data               | PASS
  DB-18   | Data retrieval: JOIN invoices + payments             | Invoice-payment relationship queryable               | Payment amounts match invoice records                | PASS
  DB-19   | Timestamp auto-generation: created_at                | Default to current timestamp                         | NOW() applied on insert                              | PASS
  DB-20   | Decimal precision: DECIMAL(10,2) for amounts         | 2 decimal places maintained                          | Rs.577.50, Rs.2520.00 stored accurately              | PASS

Conclusion:
All 20 database test cases passed. The PostgreSQL schema on Supabase maintains referential integrity
through foreign keys, enforces data validation via CHECK and NOT NULL constraints, and supports
cascading deletes for maintaining data consistency. The database is stable and ready for application integration.
