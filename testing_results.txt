Chapter 7: Testing Results


7.1 Testing Result of Sprint 1 (Database Testing)

Testing Type: Structural Testing
Database: PostgreSQL (Supabase Cloud)

During Sprint 1, the focus was on validating the database schema and ensuring data integrity across all tables. The database consists of 9 core tables: users, customers, meters, tariffs, invoices, bills, payments, and feedbacks, all created using the supabase_schema.sql migration script. Structural testing was performed to verify table creation, constraint enforcement, and data relationships.

All tables were successfully created in the Supabase PostgreSQL environment without any errors. Primary key uniqueness was validated on the customers and invoices tables, confirming that auto-increment serial IDs prevent duplicate entries. Foreign key constraints were thoroughly tested across all relationships — meters.customer_id references customers.id, invoices.customer_id references customers.id, payments.invoice_id references invoices.id, and meters.tariff_id references tariffs.id. In each case, attempts to insert records with invalid foreign key values were correctly rejected with referential integrity violation errors.

Cascading delete behavior was verified by deleting a customer record and confirming that all associated meters, invoices, and payments were automatically removed via the ON DELETE CASCADE constraint. CHECK constraints on the users.role column successfully restricted values to only 'admin' or 'customer', rejecting any other role values. The UNIQUE constraint on users.email prevented duplicate email registrations. NOT NULL constraints on critical fields such as email and password were validated by attempting null inserts, which were properly rejected.

Default values were tested for the invoices table, where the status field correctly defaults to 'Pending' and the amount_paid field defaults to 0 for new records. Seed data consisting of three tariff plans — Residential (Rs.5.50/unit), Commercial (Rs.8.00/unit), and Industrial (Rs.12.00/unit) — was inserted and verified. Sample data including 7 customers with associated meters, 15 invoices, and 12 payment records was successfully inserted, maintaining all relational integrity. JOIN queries between customers and meters, as well as between invoices and payments, returned correctly paired data. Timestamp auto-generation using NOW() was confirmed, and decimal precision using DECIMAL(10,2) accurately stored monetary values such as Rs.577.50 and Rs.2520.00.

Conclusion: All 20 database test cases passed successfully. The PostgreSQL schema on Supabase maintains referential integrity through foreign keys, enforces data validation via CHECK and NOT NULL constraints, and supports cascading deletes for maintaining data consistency. The database structure is stable, consistent, and ready for application integration.


7.2 Testing Result of Sprint 2 (Backend/Service Layer Testing)

Testing Type: Functional Testing
Technology: Supabase JavaScript Client (@supabase/supabase-js)

Sprint 2 focused on testing the backend service layer implemented in supabaseService.js, which contains over 25 functions handling all database operations. Functional testing was performed to validate CRUD operations, business logic, and data flow across all modules.

Customer management operations were tested first. Adding a new customer with valid details (first name, last name, email, contact number, and address) successfully created a record in the customers table and returned an auto-generated ID. Viewing all customers returned a complete list with properly mapped fields. Updating customer details such as contact number correctly modified the database record. Deleting a customer triggered CASCADE deletion of all associated meters, invoices, and payment records. A critical feature — automatic login credential creation — was validated by confirming that when an admin creates a new customer, a corresponding entry is inserted into the users table with the customer's email, a generated password, and the role set to 'customer'.

Meter management was tested by adding meters of different types (Smart, Digital) with associated tariff plans. Meters were correctly linked to customers via foreign keys, and viewing a customer's meters returned accurate tariff information including rate per unit and tariff description.

Invoice generation with auto-calculation was a key test area. When creating an invoice with 150 units consumed at a rate of Rs.5.50 per unit, the system correctly calculated the base amount as Rs.825.00, tax at 5% as Rs.41.25, and grand total as Rs.866.25. The combined bill and invoice creation function was also tested, confirming that both records are created simultaneously with correct linkage.

Payment processing was extensively tested. Recording a full payment equal to the grand total correctly updated the invoice status to 'Paid' and set the amount_paid field to match the grand total. Recording a partial payment correctly updated the status to 'Partially Paid' and accumulated the amount_paid field. Multiple partial payments on the same invoice were tested — two payments of Rs.300 and Rs.200 correctly resulted in a total paid amount of Rs.500.

Dashboard statistics were validated for both admin and customer roles. The admin dashboard correctly aggregated total customers, total invoices, paid and pending invoice counts, and total revenue collected. Monthly revenue aggregation properly grouped payments by month and returned sorted data for chart display. The customer dashboard accurately calculated outstanding amounts, current month consumption, next due date, and year-to-date payment totals.

Feedback functionality was tested by submitting feedback with text and star ratings, which were correctly stored with customer information. Admin view returned all feedbacks across customers, while customer view correctly filtered to show only that customer's submissions. Authorization checks were validated by attempting to access another customer's invoice, which correctly returned an 'Unauthorized' error.

Conclusion: All 25 backend test cases passed successfully. The Supabase service layer correctly handles all CRUD operations for customers, meters, invoices, and payments. Invoice generation accurately calculates base amount, tax, and grand total. The payment system properly updates invoice status on full and partial payments, and authorization checks prevent unauthorized data access.


7.3 Testing Result of Sprint 3 (System & UI Testing)

Testing Type: System Testing and Usability Testing
Deployment: Vercel (Frontend) + Supabase (Backend/Database)

Sprint 3 focused on end-to-end system testing and usability testing of the deployed application. The application was tested across multiple screen sizes — Desktop (1920x1080), Tablet (768x1024), and Mobile (375x812) — and across browsers including Google Chrome and Microsoft Edge.

Authentication testing covered all login scenarios. Admin login with valid credentials (admin@ebill.com / Admin@123) correctly redirected to the Admin Dashboard with all statistics loaded. Customer login with valid credentials redirected to the Customer Dashboard showing the billing overview. Invalid password attempts displayed a clear error message stating "Invalid email or password." Attempting to login with the wrong role (e.g., using admin credentials on the customer login) correctly showed an error. Session persistence was validated by refreshing the page after login, confirming that the user remained authenticated through localStorage. Logout functionality correctly cleared the session and redirected to the login page.

Admin panel navigation was tested across all modules. The Customers page displayed a complete table of all registered customers with search functionality that filters by name, email, or phone number in real-time. Creating a new customer through the form successfully inserted records in both the customers and users tables, with auto-generated login credentials displayed to the admin. The edit customer modal correctly pre-filled existing data and saved updates to the database. Deleting a customer showed a confirmation dialog and removed the customer along with all related data.

The Billing page displayed payment statistics (total payments, amount collected, pending count) in stat cards and showed a complete payment history table. The Create Invoice modal allowed selecting a customer, then their meter, entering units consumed, and automatically calculating the total with tax. The Feedback page displayed all customer submissions with star ratings and supported filtering by rating level and text search.

The Admin Dashboard revenue chart was tested with data spanning multiple months. The bar chart correctly scaled bars proportionally based on payment amounts, with the highest month showing the tallest bar and amounts displayed above each bar. Month labels (Feb, Mar, Apr, May) were clearly visible below.

Customer panel testing covered the complete billing workflow. The Dashboard displayed summary cards showing outstanding amount, current month consumption, next due date, and year-to-date payments. The active bill section showed the most recent unpaid invoice with a detailed breakdown and a "View & Pay" button. The Bills list page showed all invoices with color-coded status badges — green for Paid, amber for Pending, and orange for Partially Paid. The Bill detail page displayed a complete invoice breakdown including base amount, tax, grand total, amount paid, and outstanding balance, along with meter information and payment history.

Payment workflow was tested end-to-end. Making a partial payment through the payment modal correctly recorded the payment, updated the invoice's amount_paid, and changed the status to "Partially Paid." Making a subsequent payment for the remaining balance correctly updated the status to "Paid." The payment modal supported multiple payment modes (UPI, Card, Net Banking, Cash) and optional transaction reference numbers.

The feedback system was tested from the customer side. Submitting feedback with a star rating and text comment correctly stored the entry and displayed it in the feedback history. Star ratings were interactive with hover effects.

Theme switching was validated for both dark and light modes. Toggling the theme correctly updated all components — backgrounds, text colors, borders, cards, buttons, and form inputs — to the corresponding palette. The dark theme used a deep navy background (#0a0a1a) with subtle white opacity borders, while the light theme used clean whites and grays. Theme preference persisted across page refreshes via localStorage.

Responsive design was tested at three breakpoints. At desktop resolution (1920px), the full layout displayed with the navigation bar and properly spaced content. At tablet resolution (768px), cards stacked into fewer columns and tables became horizontally scrollable. At mobile resolution (375px), the navigation collapsed into a hamburger menu, cards stacked vertically, and all touch targets were appropriately sized.

Edge cases were handled gracefully throughout the application. Empty states showed friendly messages with icons when no data was available (e.g., "No payments recorded yet"). Loading states displayed spinner animations during data fetching. Error states showed clear messages in styled alert boxes. Cross-browser testing confirmed full functionality in both Chrome and Edge.

Conclusion: All 35 system and UI test cases passed successfully. The application provides a seamless user experience across both admin and customer roles. The UI is consistent in both dark and light themes, responsive across all device sizes, and handles edge cases gracefully with proper loading states, error messages, and empty state displays. The authentication system correctly manages sessions and role-based access control.


7.4 Testing Summary

A total of 80 test cases were executed across three sprints. Sprint 1 (Database Structural Testing) included 20 test cases, all of which passed, validating the PostgreSQL schema, constraints, and data integrity. Sprint 2 (Backend Functional Testing) included 25 test cases, all of which passed, confirming the correctness of CRUD operations, invoice calculations, payment processing, and authorization. Sprint 3 (System and UI Testing) included 35 test cases, all of which passed, verifying the complete user workflow, responsive design, theme switching, and cross-browser compatibility. The overall pass rate across all sprints is 100%.

Technologies validated include PostgreSQL on Supabase Cloud for the database, the Supabase JavaScript Client for serverless backend operations, React.js with TailwindCSS for the frontend, Vercel for deployment with CI/CD via GitHub, and custom table-based authentication with localStorage session management.
